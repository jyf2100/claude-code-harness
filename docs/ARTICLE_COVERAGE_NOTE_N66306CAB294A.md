# note記事「Claude Code Webを10並列で回す…」実装カバレッジ検証

対象記事: https://note.com/jujunjun110/n/n66306cab294a

## 検証方針

記事の主要トピックを以下6カテゴリに整理し、ハーネス実装の有無を判定した。

- **実装済み**: リポジトリ内に機能・仕組みが確認できる
- **部分実装**: 近い仕組みはあるが、記事の要件を満たし切らない
- **未実装**: 該当機能が確認できない（または設計のみ）

## 詳細評価

### 1) 並列実装オーケストレーション
**判定: 実装済み**

確認点:
- `/work --full --parallel N` のフルサイクル（実装→レビュー→改善→commit）
- 依存グラフに基づく並列グループ判定
- `--isolation worktree` によるタスク分離

補足:
- 記事の「10並列」をそのまま固定実装しているわけではないが、`--parallel` 指定と上限ルールにより同等の並列運用コンセプトは実装済み。

### 2) 設計→レビュー→並列実行の運用フロー
**判定: 実装済み**

確認点:
- READMEで中核フローを `/plan-with-agent` → `/work` → `/harness-review` として定義
- `/work all` による Plan→Parallel→Review→Commit の統合導線あり

### 3) ガードレール（危険操作制御）
**判定: 実装済み**

確認点:
- `.git`, `.env`, key/secrets など保護パスの拒否
- `sudo` 拒否
- `rm -rf` / `git push` の確認制御
- `git commit` 前レビュー要求（条件付き拒否）

### 4) ガードレール（品質チェック自動化）
**判定: 部分実装**

実装済み側:
- 品質判定ゲート（テンプレート/レビュー運用）
- CI セットアップ機能（lint/typecheck/test/build の雛形生成）

不足側（記事との差分）:
- 記事で強調される `dependency-cruiser` / `Knip` の標準同梱・既定実行は確認できず
- pre-push での `lint && typecheck` 強制運用は、このリポジトリでは既定フックとしては未提供（`.githooks` は pre-commit のみ）
- テスト改ざん品質ガードは設計書が中心で、ドキュメント上は「実装オプション」扱い

### 5) CLAUDE.md + ルール分割で文脈注入
**判定: 実装済み**

確認点:
- `CLAUDE.md`, `AGENTS.md`, 各種 `templates/rules/*.md.template` を配布
- スキル・ルール分割で「必要時に必要情報を読む」構成

補足:
- 記事の「slash command + CLAUDE.md で情報を構造化」の思想にかなり近い。

### 6) レイヤー設計(DDD/BC分離)の強制
**判定: 未実装（テンプレ依存）**

確認点:
- ハーネスは汎用オーケストレーション基盤であり、対象アプリのディレクトリ/レイヤー規約（例: BC別 contexts, domain/application/infrastructure の厳格分離）を自動強制する機能は確認できない
- 必要ならプロジェクトごとに rules/CI へ追加して実現する前提

---

## 総評

記事の中核である **「並列実行を回すための運用ハーネス」** は、
このリポジトリで **高い再現度（概ね 70〜80%）** で実装されている。

一方で、記事で品質担保の主力として挙げられている
**依存方向の静的制約（dependency-cruiser）や Knip まで含めた固定ガードレール一式** は、
ハーネス本体のデフォルトではなく、プロジェクト側カスタマイズ領域。

したがって結論は次の通り:

- **並列開発の実行基盤**: ほぼ実装済み
- **記事と同等の品質ガードを即時再現**: 追加設定が必要
