# SaaS 视频最佳实践

SaaS 介绍视频的最佳实践集。
提供根据视频目的·漏斗选择最佳构成的指南。

---

## 基本原则

### 1. 在画面之前先决定"消除谁的什么痛点"

不是功能介绍，而是设计成在最短时间内创造观众的痛点→解决的确定感。

**风险**: 目标模糊会导致谁都无法被打动

### 2. 在最初几秒抛弃多余的仪式

开头长时间显示 Logo 和演出会导致流失率上升，因此立即进入正题。
重视声音·正题开始的速度。

### 3. 长度由目的决定

一般而言越长参与率越低，但最佳长度取决于目的。

| 长度 | 用途 |
|------|------|
| 1-2分 | 开头流失较少 |
| 5-10分 | 较长的讲解 |

**结论**: 长视频的开头设计至关重要

### 4. CTA 不要只放在最后

为防止观众中途流失，CTA 也要放在中间。

### 5. 易看的视频是卫生管理

**优先级**: 音质 > 画面可读性 > 节奏 > 外观

**重要**: 音质差会立即流失

### 6. 字幕·转录文本是必须的

字幕是必须条件，不仅是自动生成，修正也是必要的。

### 7. 长视频要有章节

视频观众想要跳着看，所以要添加章节（目录）。

## 按漏斗分类的视频类型比较

| 目的（漏斗） | 视频类型 | 时长参考 | 构成核心 | 主要 KPI |
|------------------|------------|----------|----------|---------|
| 认知~兴趣 | 超短视频预告 | 30~90秒 | 痛点→结果（未来）→"继续看这里" | 观看维持/CTR |
| 兴趣→考虑 | Intro（概要演示） | 2~3分 | 1 个用例最短完整展示 | 演示申请/试用开始 |
| 考虑→确信 | Demo（短视频销售） | 2~5分 | 先消除反对意见的萌芽 | 商谈化率/回复率 |
| 确信→决策 | Walkthrough / Webinar | 5~30分 | 实际运营流程+证据 | CVR/咨询 |
| 扩展·效率化 | 混合演示 | 录制+Q&A | 录制标准化→直播个性化 | 成约率/工时削减 |
| 持续·活用 | 引导入门 | 30秒~数分（分小段） | 最短的快速胜利→Aha | 激活/持续率 |
| 支持 | How-to/故障 | 1~5分 | 1 个视频 1 个目的 | 咨询削减/自解决率 |

## 按类别指南

### LP/广告用：短视频介绍（30~90秒）

**目的**: 认知~兴趣漏斗

**构成大纲**:
- 0–5秒: 痛点 or 期望结果
- 5–20秒: 目标用户和承诺
- 20–60秒: 象征性工作流 1 个
- 60–90秒: 下一步行动

**应避免的陷阱**: 功能罗列·抽象词汇连发

### 考虑用：2~3 分 Intro 演示

完整展示 1 个用例，避免专业术语，边分解边说明。

**推荐事项**:
- 开头简短进入正题
- 选择最打动人的 1 个案例
- 中途也设置 CTA

### 销售用：2~5 分演示

促使购买决策进入下一步的短视频销售用视频。

**策略**:
- 先消除反对意见前 3 名
- 1 视频=1 行业/职业

### 决策用：15~30 分演示

更深入理解和促进决策的长视频。章节设计必须。

**关键点**:
- 基本流程
- 反对意见处理
- 管理/安全说明

### 引导入门用

设计用户首次到达成功体验（Aha）。

**指南**:
- 1 个视频不要教全部
- 按角色分支视频

### 支持/帮助视频

不要让视频成为唯一答案，与文本并用。

**必须要素**:
- 视频+步骤文本
- 字幕+全文转录文本

## 模板

### 90 秒打动人的介绍模板

**用途**: LP/广告、认知~兴趣漏斗

| 时间 | 内容 | 帧数(30fps) |
|------|------|------------------|
| 0:00–0:05 | 痛点 or 期望结果 | 0-150 |
| 0:05–0:15 | 目标用户和承诺 | 150-450 |
| 0:15–0:55 | 象征性工作流 | 450-1650 |
| 0:55–1:10 | 差异化依据 | 1650-2100 |
| 1:10–1:30 | CTA | 2100-2700 |

**合计**: 90秒 = 2700帧

### 3 分 Intro 演示模板

**用途**: 考虑用、兴趣→考虑漏斗

| 时间 | 内容 |
|------|------|
| 0:00–0:10 | 结论+痛点 |
| 0:10–0:30 | 用例声明 |
| 0:30–2:20 | 实际画面完整展示 |
| 2:20–2:50 | 消除一个常见顾虑 |
| 2:50–3:00 | CTA |

### 20 分决策用演示

**用途**: 确信→决策漏斗

| 时间 | 内容 |
|------|------|
| 0:00–1:00 | 对象和课题 |
| 1:00–8:00 | 基本流程 |
| 8:00–12:00 | 反对意见前 2 名 |
| 12:00–15:00 | 管理/安全 |
| 15:00–20:00 | 成功案例+CTA |

## 制作检查清单

### 收录前

- [ ] 脚本：精简说话内容
- [ ] 演示环境：关闭通知、去除个人信息
- [ ] 画面：UI 文字可读的缩放

### 收录中

- [ ] 音频：清晰度最优先
- [ ] 照明/拍摄：最低限度
- [ ] 节奏：操作犹豫的瞬间全部剪切

### 发布时

- [ ] 字幕：务必质量修正
- [ ] 转录文本：用于搜索和略读
- [ ] 长视频必须添加章节

## 常见失败模式

| 失败 | 影响 |
|------|------|
| 目标用户不明确 | 谁都无法被打动 |
| Logo·公司介绍过长 | 开头流失 |
| 功能全收录 | 焦点模糊 |
| 无字幕/粗糙 | 可访问性降低 |
| 只放视频就满足 | 无法测量效果 |
| CTA 只在最后 | 中途流失无法看到 |

## 推荐 3 个视频套装

最短取得成果的基本套装:

1. **90 秒预告** - 获取认知
2. **3 分 Intro 演示** - 促进考虑
3. **15~25 分决策用演示** - 支持成交

**成果**: 填补获取→考虑→决策的信息缺口

## Harness 中的应用

### 视频类型自动判定

| Harness 检测条件 | 推荐视频类型 | 模板 |
|-----------------|----------------|--------------|
| 新项目 | LP/广告用短视频 | 90 秒预告 |
| 检测到 UI 变更 | Intro 演示 | 3 分模板 |
| CHANGELOG 更新 | 发布说明 | 重视 Before/After |
| 大规模结构变更 | 架构讲解 | 演示 |

### 场景构成指南

#### 短视频（30-90秒）时

```
HookScene (3秒) → ProblemPromise (7秒) → WorkflowDemo (40-60秒) → Differentiator (10秒) → CTA (10秒)
```

#### Intro 演示（2-3分）时

```
Hook (10秒) → UseCase声明 (20秒) → 实际画面演示 (110秒) → 消除顾虑 (30秒) → CTA (10秒)
```

#### 演示（15-30分）时

```
对象和课题 (1分) → 基本流程 (7分) → 反对意见处理 (4分) → 管理/安全 (3分) → 成功案例+CTA (5分)
```

## Remotion 实现规则

### 动画

| 规则 | 原因 |
|--------|------|
| `useCurrentFrame()` 必须 | 禁止 CSS animations，使用 Remotion 的帧控制 |
| `spring({ damping: 200 })` | 平滑运动 |
| `interpolate()` + `extrapolateRight: 'clamp'` | 防止值失控 |
| 打字机效果用 `text.slice(0, charCount)` | 不推荐按字符更改 opacity |

### 音频

| 规则 | 原因 |
|--------|------|
| `Audio` 从 `@remotion/media` import | `Html5Audio` 不推荐 |
| 音频开始 = 场景开始 + 30f（1秒等待） | 幻灯片切换后视觉稳定再开始音频 |
| 场景长度 = 30f + 音频长度 + 20f余量 | 过渡前音频结束 |
| 用 `ffprobe` 事先确认音频长度 | 场景设计的前提信息 |

### TransitionSeries

| 规则 | 原因 |
|--------|------|
| 过渡长度推荐 15f（0.5秒） | 自然切换 |
| 场景开始 = 前一场景开始 + 前一场景长度 - 过渡长度 | 考虑重叠 |
| 音频在过渡开始前结束 | 切换中的音频会产生违和感 |

### 场景长度计算示例

```
音频长度确认（30fps 换算）:
  hook: 4.0秒 = 121帧
  problem: 12.0秒 = 360帧

场景长度计算:
  hook: 30(等待) + 121(音频) + 24(余量) = 175帧
  problem: 30(等待) + 360(音频) + 25(余量) = 415帧

场景开始帧（过渡 15f）:
  hook: 0
  problem: 175 - 15 = 160

音频开始时机:
  hook: 0 + 30 = 30
  problem: 160 + 30 = 190
```
