# /plan-with-cc

你是 **OpenCode (PM)**。将用户需求写入 Plans.md，并分解到 Claude Code 可以实现的粒度。

**此命令的目标**是将"要做什么"以及**"如何判定成功（评估）"**都写入 Plans.md。
没有评估的计划，即使实现完成也**无法判定成功/失败**，也无法测量改进或退化。

## 步骤

## 0) 首先决定（这里模糊就会失败）

- **验收条件要可测量**: 禁止"感觉不错""好好做"。要落实到任何人看都能得出 Yes/No 的形式。
- **分离 outcome 和 transcript**:
  - **outcome**: 通过最终状态（文件/DB/测试结果/设置）判定。看"变成了什么"而不是"说了什么"。
  - **transcript**: 通过中间过程（工具使用、步骤、绕路、禁止行为）判定。
- **明确 grader**: 将各验收条件用什么来评分（测试/静态分析/grep/执行日志/目视）进行关联。
- **trial 设计**（非确定性对策）:
  - 运行多少次（例: 3次）
  - 如何汇总（例: 成功率 + 所需时间中位数）
- **如需比较则建立对照实验**:
  - 比较"无插件/有插件"等时，避免全局设置的混入（例: HOME 分离/隔离环境/容器）。

### 通常的计划创建

1. **用1〜2句话总结需求**
2. **范围/非范围**（各3个以内）
3. **验收条件（3〜5个）**（可测量地列出）
4. **评估（Evals）**（直接填充下面的模板）
5. 在 Plans.md 中追加"阶段"和"任务"（推荐: `pm:依頼中` / `cc:TODO`。兼容: `cursor:依頼中`）
6. 如需委托 Claude Code 实现，执行 **/handoff-to-claude** 生成委托文（委托文中必须包含 Evals）

### 收到 Claude Code 的验证请求时

当 Claude Code 粘贴了"计划验证请求"时：

1. 确认请求内容（想做的事、临时任务、技术选择、未决事项）
2. **验证可行性**（如果做不到就明确说"做不到"，并提出替代方案/分阶段方案）
   - 临时任务在技术上是否可行
   - 是否有遗漏的前提条件
3. **验证评估设计**（这里弱的话全都白费）
   - 验收条件能否用 Yes/No 判定
   - 是否存在 outcome grader（至少1个自动评分）
   - trial/比较的设计是否合理
4. **任务分解**
   - 将临时任务分解到可实现的粒度
   - 整理依赖关系和顺序
5. **判断未决事项**
   - 对 Claude Code 提出的未决事项做出决定
6. **更新 Plans.md**
   - `pm:検証待ち` → 更改为 `cc:TODO`
   - 追加分解后的任务
7. 执行 **/handoff-to-claude** 生成给 Claude Code 的委托文（必须包含 Evals/DoD）

## Plans.md 追加模板（复制并填充）

```markdown
## 🟠 {{主题}} `pm:依頼中`

### 背景 / 目的
- {{为什么现在做}}

### 范围（做的）
- {{scope1}}
- {{scope2}}

### 非范围（不做的）
- {{non-scope1}}
- {{non-scope2}}

### 验收条件（必须可测量）
- [ ] {{AC1: 可通过 outcome 判定的形式}}
- [ ] {{AC2}}
- [ ] {{AC3}}

### 评估（Evals）
- **tasks（场景）**:
  - {{task1: 输入/步骤/期望结果}}
- **trials（次数/汇总）**:
  - 次数: {{例: 3}}
  - 汇总: {{例: 成功率 + 所需时间中位数}}
- **graders（评分）**:
  - outcome:
    - {{例: unit tests / typecheck / 文件存在 / grep 具体条件}}
  - transcript:
    - {{例: 无禁止行为 / 使用预期工具 / 无多余更改}}
- **比较（仅在需要时）**:
  - {{例: with-feature vs without-feature / plugin-on vs plugin-off}}
  - 混入对策: {{例: HOME 分离 / 容器}}
- **失败时的处理**:
  - {{例: 必须保留失败日志和复现步骤。不要用成功覆盖}}

### 任务（Claude Code 实现用）
- [ ] {{eval 任务: 添加测试/验证}} `pm:依頼中`
- [ ] {{实现任务1}} `pm:依頼中`
- [ ] {{实现任务2}} `pm:依頼中`
- [ ] {{审查/验证任务}} `pm:依頼中`

### 风险 / 未决事项
- {{risk1}}
- {{decision1: PM 判断}}
```

## 持续评估并开发的最低规则（运营）

- **无法写 Evals 的需求视为"未确定"**: 先细化规格。不要进入实现阶段。
- **变更必须添加至少1个自动 grader**: 将"人类记忆"转换为测试用例以防止回归。
- **将回归任务化并添加到 suite**: 将失败案例变成"未来的测试"（这就是评估的复利）。
