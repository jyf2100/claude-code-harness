# General Principles

使用 claude-code-harness 时的基本原则。适用于所有工作流。

---

## 安全原则

### 1. 确认后再更改

编辑文件前，务必确认以下内容：

- **用 Read 工具确认内容**：理解现有代码后再更改
- **把握影响范围**：考虑变更对其他文件的影响
- **考虑备份**：重要变更前确认 git status

### 2. 以最小差异编辑

```
❌ 错误示例：重写整个文件
✅ 正确示例：用 Edit 工具只修改必要部分
```

### 3. 尊重配置文件

遵循 `claude-code-harness.config.json` 的设置：

- `safety.mode`：dry-run / apply-local / apply-and-push
- `paths.protected`：不更改受保护路径
- `paths.allowed_modify`：只更改允许的路径

---

## 工作原则

### 1. 始终更新 Plans.md

- 任务开始时：`cc:TODO` → `cc:WIP`
- 任务完成时：`cc:WIP` → `cc:完成`
- 阻塞时：添加 `blocked` 并注明原因

### 2. 分阶段推进

```
1. 调查・理解 → 2. 计划 → 3. 实现 → 4. 确认 → 5. 报告
```

### 3. 出错时的应对

- 最多 3 次自动重试
- 无法解决时升级（报告）
- 明确记录错误内容和尝试过的对策

---

## 沟通原则

### VibeCoder 对应

确保无技术知识也能理解：

- **避免专业术语**：或添加说明
- **提示下一步行动**：「接下来请说〇〇」
- **可视化进度**：明确已完成和剩余内容

### PM（Cursor）协作

- **在 Plans.md 共享状态**：作为单一来源维护
- **完成报告用 `/handoff-to-cursor`**：遵循格式
- **守住范围**：超出委托范围的工作要确认

---

## 禁止事项

1. **直接部署到生产环境**（只到 staging）
2. **硬编码机密信息**（使用 .env）
3. **更改受保护路径**（.github/, secrets/ 等）
4. **未经用户确认的破坏性操作**（rm -rf 等）
