# SaaS 视频最佳实践

SaaS 介绍视频的最佳实践集合。
提供根据视频目的和漏斗选择最佳结构的指南。

---

## 基本原则

### 1. 在展示画面之前先确定"消除谁的什么痛点"

不是功能介绍，而是设计成在最短时间内建立观众痛点→解决方案确信的装置。

**风险**: 目标模糊会导致谁都不感兴趣

### 2. 最初的几秒放弃多余的仪式

开头长时间展示标志和演出会导致跳出率上升，要尽快进入正题。
重视声音和正题开始的速度。

### 3. 长度由目的决定

一般来说越长参与率越低，但最佳长度取决于目的。

| 长度 | 用途 |
|------|------|
| 1-2分钟 | 开头跳出较少 |
| 5-10分钟 | 较长的说明 |

**结论**: 长视频开头设计是关键

### 4. CTA 不要只放在最后

为防止观众中途跳出，CTA 也要放在中间。

### 5. 易于观看的视频是基础卫生

**优先级**: 音质 > 画面可读性 > 节奏 > 外观

**重要**: 声音差会直接导致跳出

### 6. 字幕和转录文本是必须的

字幕是必要条件，不仅自动生成还需要修正。

### 7. 长视频需要章节

视频观众希望跳着看，所以要添加章节（目录）。

## 漏斗各视频类型对比

| 目的（漏斗） | 视频类型 | 长度参考 | 结构核心 | 主要KPI |
|------------------|------------|----------|----------|---------|
| 认知～兴趣 | 超短片预告 | 30～90秒 | 痛点→结果（未来）→"后续在这里" | 视听维持/CTR |
| 兴趣→考虑 | Intro（概要演示） | 2～3分钟 | 1个用例最短完播 | 演示申请/试用开始 |
| 考虑→确信 | Demo（短片销售） | 2～5分钟 | 先消除反驳苗头 | 商谈转化率/回复率 |
| 确信→决策 | Walkthrough / Webinar | 5～30分钟 | 实际使用流程+证据 | CVR/咨询 |
| 扩展·效率化 | 混合演示 | 录制+Q&A | 录制标准化→直播个性化 | 成约率/工时削减 |
| 持续·活用 | 入门引导 | 30秒～数分钟（分割） | 最短的快速成功→Aha | 激活/持续率 |
| 支持 | How-to/故障 | 1～5分钟 | 1个视频1个目的 | 咨询削减/自行解决率 |

## 各类别指南

### LP/广告用：短片介绍（30～90秒）

**目的**: 认知～兴趣漏斗

**结构大纲**:
- 0–5秒: 痛点 or 期望的结果
- 5–20秒: 目标用户和承诺
- 20–60秒: 象征性工作流1个
- 60–90秒: 下一步行动

**应避免的陷阱**: 罗列功能·抽象词汇连发

### 考虑用：2～3分钟 Intro 演示

让1个用例完播，避免专业术语，边分解边说明。

**推荐事项**:
- 开头简短快速进入正题
- 选择最吸引人的1个案例
- 中间也设置 CTA

### 销售用：2～5分钟演示

促进购买决策进入下一步的短片销售视频。

**战略**:
- 先消除前3个反驳
- 1个视频=1个行业/职业

### 决策用：15～30分钟演示

促进更深入理解和决策的长视频。必须设计章节。

**关键点**:
- 基本流程
- 反驳处理
- 管理/安全说明

### 入门引导用

设计让用户达到首次成功体验（Aha）。

**指南**:
- 不要1个视频教全部
- 按角色分支视频

### 支持/帮助视频

不要让视频成为唯一答案，与文本并用。

**必要元素**:
- 视频+步骤文本
- 字幕+全文转录

## 模板

### 90秒吸引人介绍模板

**用途**: LP/广告、认知～兴趣漏斗

| 时间 | 内容 | 帧数(30fps) |
|------|------|------------------|
| 0:00–0:05 | 痛点 or 期望的结果 | 0-150 |
| 0:05–0:15 | 目标用户和承诺 | 150-450 |
| 0:15–0:55 | 象征性工作流 | 450-1650 |
| 0:55–1:10 | 差异化依据 | 1650-2100 |
| 1:10–1:30 | CTA | 2100-2700 |

**合计**: 90秒 = 2700帧

### 3分钟 Intro 演示模板

**用途**: 考虑用、兴趣→考虑漏斗

| 时间 | 内容 |
|------|------|
| 0:00–0:10 | 结论+痛点 |
| 0:10–0:30 | 用例声明 |
| 0:30–2:20 | 实际画面完播 |
| 2:20–2:50 | 消除1个常见担忧 |
| 2:50–3:00 | CTA |

### 20分钟决策用演示

**用途**: 确信→决策漏斗

| 时间 | 内容 |
|------|------|
| 0:00–1:00 | 对象和课题 |
| 1:00–8:00 | 基本流程 |
| 8:00–12:00 | 反驳前2个 |
| 12:00–15:00 | 管理/安全 |
| 15:00–20:00 | 成功案例+CTA |

## 制作检查清单

### 录制前

- [ ] 台本：删减到极致
- [ ] 演示环境：关闭通知、去除个人信息
- [ ] 画面：UI文字可读的缩放

### 录制中

- [ ] 声音：清晰度最优先
- [ ] 灯光/画面：最低限度
- [ ] 节奏：犹豫的瞬间全部剪掉

### 发布时

- [ ] 字幕：必须进行质量修正
- [ ] 转录文本：用于搜索和浏览
- [ ] 长视频必须添加章节

## 常见失败模式

| 失败 | 影响 |
|------|------|
| 目标受众模糊 | 谁都不感兴趣 |
| 标志·公司介绍太长 | 开头跳出 |
| 功能全包含 | 焦点模糊 |
| 无字幕/粗糙 | 无障碍性降低 |
| 只放视频就满足 | 无法测量效果 |
| CTA只在最后 | 中途跳出看不到 |

## 推荐3套组合

最短获得成果的基础组合:

1. **90秒预告** - 获取认知
2. **3分钟 Intro 演示** - 促进考虑
3. **15～25分钟决策用演示** - 支持成约

**成果**: 填补获取→考虑→决策的信息空白

## Harness 中的活用

### 视频类型自动判定

| Harness检测条件 | 推荐视频类型 | 模板 |
|-----------------|----------------|--------------|
| 新项目 | LP/广告用短片 | 90秒预告 |
| UI变更检测 | Intro 演示 | 3分钟模板 |
| CHANGELOG 更新 | 发布说明 | 重视 Before/After |
| 大规模结构变更 | 架构说明 | 演示 |

### 场景结构指南

#### 短片（30-90秒）的情况

```
HookScene (3秒) → ProblemPromise (7秒) → WorkflowDemo (40-60秒) → Differentiator (10秒) → CTA (10秒)
```

#### Intro 演示（2-3分钟）的情况

```
Hook (10秒) → 用例声明 (20秒) → 实际画面演示 (110秒) → 消除担忧 (30秒) → CTA (10秒)
```

#### 演示（15-30分钟）的情况

```
对象和课题 (1分) → 基本流程 (7分) → 反驳处理 (4分) → 管理/安全 (3分) → 成功案例+CTA (5分)
```

## Remotion 实现规则

### 动画

| 规则 | 理由 |
|--------|------|
| `useCurrentFrame()` 必须 | 禁止 CSS animations，使用 Remotion 的帧控制 |
| `spring({ damping: 200 })` | 平滑的运动 |
| `interpolate()` + `extrapolateRight: 'clamp'` | 防止数值失控 |
| 打字机效果使用 `text.slice(0, charCount)` | 不推荐逐字符的 opacity 修改 |

### 音频

| 规则 | 理由 |
|--------|------|
| `Audio` 从 `@remotion/media` import | `Html5Audio` 已废弃 |
| 音频开始 = 场景开始 + 30f（等待1秒） | 幻灯片切换后视觉稳定再开始音频 |
| 场景长度 = 30f + 音频长度 + 20f余量 | 过渡前音频结束 |
| 用 `ffprobe` 事先确认音频长度 | 场景设计的前提信息 |

### TransitionSeries

| 规则 | 理由 |
|--------|------|
| 过渡长度推荐 15f（0.5秒） | 自然的切换 |
| 场景开始 = 前场景开始 + 前场景长度 - 过渡长度 | 考虑重叠 |
| 音频在过渡开始前结束 | 切换中的音频会感到违和 |

### 场景长度计算例

```
音频长度确认（30fps换算）:
  hook: 4.0秒 = 121帧
  problem: 12.0秒 = 360帧

场景长度计算:
  hook: 30(等待) + 121(音频) + 24(余量) = 175帧
  problem: 30(等待) + 360(音频) + 25(余量) = 415帧

场景开始帧（过渡15f）:
  hook: 0
  problem: 175 - 15 = 160

音频开始时机:
  hook: 0 + 30 = 30
  problem: 160 + 30 = 190
```
