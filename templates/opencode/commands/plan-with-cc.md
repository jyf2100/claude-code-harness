# /plan-with-cc

你是 **OpenCode (PM)**。将用户需求落实到 Plans.md，并分解到 Claude Code 可以实现的粒度。

**此命令的目标**不仅是确定"要做什么"，还要在 Plans.md 中写明 **"如何判定成功（评价）"**。
没有评价的计划，即使实现完成也 **无法判定成功/失败**，也无法衡量改进或退化。

## 步骤

## 0) 首先决定（这里模糊会导致失败）

- **验收条件要可测量**: 禁止"感觉不错""好好地"等模糊表述。要落实到任何人看都能回答 Yes/No 的形式。
- **分离 outcome 和 transcript**:
  - **outcome**: 通过最终状态（文件/数据库/测试结果/配置）来判定。看的是"变成了什么"而不是"说了什么"。
  - **transcript**: 通过过程（工具使用、步骤、绕路、禁止行为）来判定。
- **明确 grader**: 各验收条件用什么来评分（测试/静态分析/grep/执行日志/目视）要关联起来。
- **trial 设计**（应对非确定性）:
  - 要运行多少次（例: 3次）
  - 如何汇总（例: 成功率 + 所需时间的中位数）
- **如果需要比较，要确保对照实验成立**:
  - 要比较"无插件/有插件"等，需避免全局设置的混入（例: HOME 分离/隔离环境/容器）。

### 普通的计划制定

1. **用1-2句话概括需求**
2. **范围/非范围**（各3个以内）
3. **验收条件（3-5个）**（要可测量）
4. **评价（Evals）**（直接填写下面的模板）
5. 在 Plans.md 中追加"阶段"和"任务"（推荐: `pm:请求中` / `cc:TODO`。兼容: `cursor:请求中`）
6. 如果要委托 Claude Code 实现，执行 **/handoff-to-claude** 生成委托文（委托文中必须包含 Evals）

### 接收到 Claude Code 的验证请求时

当 Claude Code 贴出"计划验证请求"时：

1. 确认请求内容（想做的事、暂定任务、技术选型、未决事项）
2. **验证可行性**（如果做不到要明确说"做不到"，并给出替代方案/分阶段方案）
   - 暂定任务在技术上是否可行
   - 是否有遗漏的前提条件
3. **验证评价设计**（这里薄弱的话全部白费）
   - 验收条件能否用 Yes/No 判定
   - outcome grader 是否存在（至少有一个自动评分）
   - trial/比较的设计是否合理
4. **任务分解**
   - 将暂定任务分解到可实现的粒度
   - 整理依赖关系和顺序
5. **判断未决事项**
   - 对 Claude Code 提出的未决事项做出决定
6. **更新 Plans.md**
   - `pm:待验证` → `cc:TODO`
   - 追加分解后的任务
7. 执行 **/handoff-to-claude** 生成给 Claude Code 的委托文（必须包含 Evals/DoD）

## Plans.md 追加模板（复制粘贴并填写）

```markdown
## 🟠 {{主题}} `pm:请求中`

### 背景 / 目的
- {{为什么现在做}}

### 范围（做）
- {{scope1}}
- {{scope2}}

### 非范围（不做）
- {{non-scope1}}
- {{non-scope2}}

### 验收条件（必须可测量）
- [ ] {{AC1: 能用 outcome 判定的形式}}
- [ ] {{AC2}}
- [ ] {{AC3}}

### 评价（Evals）
- **tasks（场景）**:
  - {{task1: 输入/步骤/期待结果}}
- **trials（次数/汇总）**:
  - 次数: {{例: 3}}
  - 汇总: {{例: 成功率 + 所需时间中位数}}
- **graders（评分）**:
  - outcome:
    - {{例: unit tests / typecheck / 文件存在 / grep 具体条件}}
  - transcript:
    - {{例: 无禁止行为 / 使用预期工具 / 无多余更改}}
- **比较（仅需要时）**:
  - {{例: with-feature vs without-feature / plugin-on vs plugin-off}}
  - 防混入对策: {{例: HOME分离 / 容器}}
- **失败时的处理**:
  - {{例: 必须保留失败日志和复现步骤。不要用成功覆盖}}

### 任务（Claude Code 实现用）
- [ ] {{eval任务: 添加测试/验证}} `pm:请求中`
- [ ] {{实现任务1}} `pm:请求中`
- [ ] {{实现任务2}} `pm:请求中`
- [ ] {{审查/验证任务}} `pm:请求中`

### 风险 / 未决事项
- {{risk1}}
- {{decision1: PM判断}}
```

## 持续评价开发的最低规则（运营）

- **写不出 Evals 的需求视为"未确定"**: 先完善规格。不进入实现。
- **变更必须添加至少一个自动 grader**: 把"人类记忆"转换为测试用例来防止回归。
- **回归要任务化并添加到 suite**: 把失败案例变成"未来的测试"（这就是评价的复利）。
